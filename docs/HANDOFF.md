## Facts To Preserve

- `PetService` remains the single source of truth for pet state.
- Pet grounded/home behavior remains host-owned movement policy, configured from Control Center rather than pet packs or AI rules.
- New user-facing configuration belongs in Control Center.
- API keys must stay out of the renderer.
- AI chat provider settings use a draft/active split. Saving and testing must go through main-process `AiService`, return sanitized structured diagnostics, and must not expose API keys, Authorization headers, prompts, or credentialed Base URLs to renderer/plugin contexts.
- The Control Center AI pane now treats chat and image Provider configuration as the primary first-screen model-settings surface: one `模型 Provider` section opens by default with `聊天模型` and `图片模型` capability cards, secondary AI sections stay collapsed until expanded, and both Provider cards explicitly explain that local and cloud gateways share one OpenAI-compatible contract while secrets stay host-owned.
- Image generation for Creator Studio is host-owned at the provider boundary. Control Center now exposes one OpenAI-compatible image Provider surface with presets, optional `/models` discovery, provider-aware compatibility hints, and health checks; `ImageGenerationModelService` owns provider calls and output writes, and Creator Studio consumes host-mediated outputs instead of receiving provider credentials.
- Creator Studio also has a matching host-side smoke path for that boundary: `npm run smoke:creator-studio-provider` reuses the saved image Provider config and secret, runs prompt building plus provider generation plus action-frame QA, and writes a sanitized session report without exposing raw API keys.
- A successful real AI provider smoke archive now lives under `docs/release-evidence/ai-provider-smoke/2026-06-28T11-08-10Z-openpet-gateway/`: the current OpenPet development gateway exposed `gpt-5.5` and `gpt-image-2` through `/models`, `gpt-5.5` passed chat completion smoke, image generation remained intentionally skipped to avoid implicit credit spend, and the archived report keeps the API key redacted.
- The archived real AI provider smoke evidence confirms model discovery and chat-completion reachability for the current OpenPet development gateway, but it does not prove image generation output quality, transparent-background quality, or production asset readiness.
- AI Talk also has a separate real-provider Bubble Chat acceptance smoke path through `npm run run-ai-talk-local-smoke -- --message "..."`: it reuses the saved host-side AI Talk wiring, records `bubbleAcceptance.requestId`, `providerLatencyMs`, `bubbleDispatch` visibility evidence, and a `manualAcceptanceTemplate` placeholder for later desktop-feel review, but it does not by itself prove transparent popup placement, dwell-time quality, or full human acceptance.
- A fresh archived AI Talk Bubble Chat smoke run now lives under `docs/release-evidence/ai-talk-local-smoke/2026-06-28T15-35-59-210Z/`: `gpt-5.5` returned `你好呀，我在这儿陪你～🐾`, `bubbleAcceptance.requestId` was correlated through AI Talk and Bubble Chat logs, `providerLatencyMs` was `2141`, `bubbleDispatch.petSayReceived` was `true`, `bubbleDispatch.bubbleStateVisible` was `true`, and the popup telemetry recorded `ttlMs = 9835` before auto-hide.
- A successful real Creator Studio provider smoke archive now lives under `docs/release-evidence/creator-studio-provider-smoke/2026-06-28T14-06-27-403Z/`: the saved host-owned `gpt-image-2` path passed health check, returned one source PNG in about `265s` using requested `512x512` generation constraints, and passed 16-frame action QA with zero warnings under a temporary `420000ms` timeout override.
- The archived real smoke evidence confirms the current OpenPet development gateway can satisfy that host-owned Creator Studio path, but only with a long-running image request; treat lower timeout expectations and asset-quality claims as still unproven until separately validated.
- Packaged runtime smoke evidence is archived under `docs/release-evidence/packaged-runtime/2026-06-16T14-52-13-074Z-darwin-arm64/`: it proves an unsigned macOS packaged runtime launched, rendered transparently, played built-in pack actions, and restored stable state, but `plugin-picker-evidence-linked` and `pet-picker-evidence-linked` remain pending and `invalid-package-feedback` remains blocked until a reviewed native desktop-picker smoke report is linked.
- Signed release closure evidence is archived under `docs/release-evidence/signed-release-closure/2026-06-16T15-00-00Z/`: official desktop, macOS, and Windows claims are all `not-ready`, with blockers including missing signed macOS codesign/notarization/Gatekeeper evidence, missing desktop picker evidence, unsigned packaged runtime evidence, and missing signed Windows smoke evidence.
- Release evidence remains Manual-required beyond those archived facts: Apple signing/notarization credentials, accepted Gatekeeper/notary outputs, signed Windows artifacts on real Windows, and human review of the archived evidence bundles are still required before any release-ready wording can change.
- Creator Studio's dashboard-first wizard is now the primary user-facing workflow for run draft, follow-up questions, confirmation, generation, QA review, retry/recover, prompt provenance, workflow guidance, action-lane guidance, and import handoff. Approved action/pet runs now surface a structured handoff card that tells the dashboard which Control Center plugin command to run next, while import itself remains blocked to explicit command runs so bridge tokens stay command-scoped. Provider-backed full-pet runs build a real generated atlas, write `source-image-validation.json` and `atlas-validation.json`, and must pass QA before `Import Approved Pet` can call the host bridge. The command entrypoints remain the automation and host-bridge entry path.
- Creator Studio imported follow-up routing is outcome-specific and now shared across `nextStep`, `actionLane`, `workflowGuidance.import.followUp`, and imported result review surfaces: imported action success routes reviewers to `Actions -> Trigger Proposal Inbox`, imported action handoff failures route follow-up to `Control Center -> Plugins`, and imported pet follow-up remains `OpenPet` through `Import Approved Pet`.
- Creator Studio imported review surfaces are now phase-aware: once a run is `imported`, the dashboard still shows imported result and follow-up guidance, but it no longer carries over pre-import QA blocked notices, repair controls, or retry-generation cues from the approval phase.
- Action trigger proposals are review inputs, not plugin-owned mutations. The host can currently apply `click` proposals to `clickAction`, acknowledge `manual`/`unbound`, and `random`, `state`, and `event` proposals create active host-owned durable trigger rules. Creator Studio action imports submit their trigger proposal into the Control Center Actions `Trigger Proposal Inbox`; the plugin still does not apply trigger rules directly.
- Creator Studio dashboard routes stay local to the plugin service. Command-scoped bridge tokens are not available to the dashboard service, unknown dashboard `/api/*` paths return JSON `404`, and privileged host imports/health checks still require explicit command runs.
- Extension docs must be honest: OpenPet now parses declarations, can explicitly run `entries.setup` for enabled policy-allowed local plugins, can run `entries.commands` through the JavaScript compatibility runner when `main` exists, can explicitly run declaration-only local `entries.commands` as short-lived processes with JSON stdin context, can inject short-lived bridge URL/token plus host-owned data/cache/log env vars for those declaration-only command runs, can expose bounded creator-tools action reads / validation / apply, active installed user pack metadata workflows, package-local frame inspection/import, and user-approved picker frame inspection/import through the same short-lived bridge, can explicitly open declared HTTP/HTTPS dashboards for enabled plugins, can explicitly start/stop declared local service entries, can manually check declared loopback service health endpoints, can host-manage periodic health checks for running services through Control Center, attempts best-effort process-group cleanup when stopping service entries, only reports setup/command/service stop completion after child exit confirmation, will attempt one bounded host-side force stop if the service ignores the grace-period stop request, now tries a host-owned process-tree fallback before direct child kill across service/setup/declaration-command stop paths, and can record/update/collect/run/archive bounded cleanup evidence through controlled host fixtures, generated collector helpers, runner transcripts, structured readiness reports, archive manifests, and validation-first report updates. Submission bundles can now also receive a separate structured maintainer approval record. The current docs-and-tests lockstep also pins bridge route and permission coverage, including `trigger-proposals:write` for Creator Studio review handoff and `model:image-generate` for host-managed settings, health checks, and bounded generation, while plugin-managed provider credentials stay unsupported on that host-managed generation surface. Approval remains a human review decision and not signing trust, catalog publication, runtime safety, or release readiness proof. Command, setup, and service processes are spawned without shell expansion. Services do not auto-start, setup and command entries do not run during install or enable, background checks stay opt-in and runtime-bound, picker frame import and pack metadata workflows do not imply raw filesystem grants, raw file writes, plugin-selected output paths, built-in pack edits, arbitrary pack targeting, general pet-pack writes, or universal process-tree cleanup guarantees, and OpenPet does not claim complete sandboxing for arbitrary local processes.
- `cat_anime/` structure is unchanged.
- Windows is not release-ready yet.
