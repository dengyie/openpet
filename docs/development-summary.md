## Open Work

- Windows is still not release-ready until real signed installer evidence and smoke reports are archived.
- AI provider settings UX is now implemented for chat provider configuration: active config and drafts are separated, save and test actions are visible inside the chat Provider section, save does not depend on test success, connection test results are structured and sanitized, and credentialed Base URLs are not downgraded or exposed in renderer views. Future AI work should preserve that boundary rather than reintroducing renderer-owned secrets.
- Host image-generation settings now exist for Creator Studio: Control Center and the host runtime both use one OpenAI-compatible image Provider contract with `Base URL`, `Model`, timeout/concurrency, and main-process `API Key` storage through `SecretService`; local and hosted providers are selected by changing the Provider URL rather than by exposing separate `local` / `cloud` modes. The Control Center image Provider section now includes setup guidance, OpenAI-compatible presets, editable timeout/concurrency fields, and discovered-model suggestions. Older `fixture` / `cloud` / `local` settings are read as legacy input and normalized into the unified Provider view, while new saves persist only the unified shape. Ordinary extensions still do not receive those saved provider credentials; any third-party provider secrets remain extension-owned unless a future explicit trust model is added. The remaining product work is connection history, named provider profiles, and broader failover/routing policy.
- Creator Studio dashboard-first wizard is now the primary user-facing flow on top of the existing command/service boundary: dashboard workflow summaries expose recommended next actions, pending questions use real question ids/prompts/options, failed generation states expose direct retry affordances, and command paths remain intact as the automation/test entry points.
- Action trigger proposal acceptance is implemented for reviewed proposals: `click` can update `clickAction`, while `manual` and `unbound` are acknowledged without mutating bindings. `random`, `state`, and `event` proposals still require a full host trigger-rule schema/editor before they can be applied.
